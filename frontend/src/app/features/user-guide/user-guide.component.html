User guide How to use the API Documentation Platform: where each screen is, what it does, and a typical workflow from defining an API to testing and exporting PDFs.
Uses your browser’s print dialog — choose “Save as PDF” to download a copy of this guide.
1. Before you start
  1. Backend (Spring Boot) must be running and reachable at {{ apiBaseUrl }} (configured in frontend/src/environments/environment.ts).
  2. This web app (Angular) is usually opened at http://localhost:9091 after npm start in the frontend folder.
  3. If lists fail to load, confirm the backend URL, that the server is up, and that the browser is not blocking requests (CORS is configured for typical local dev origins).
2. Where things are (left navigation)
Menu item Route Purpose
API List /apis See all saved API definitions, search and filter, open view/edit/test, delete.
API Builder /builder Create a new API or edit one: base URL, HTTP method, path template, description, request field tree.
API Tester /tester Pick an API, build the JSON body from the saved schema, then validate URL/body or execute the real call.
PDF Viewer /pdf Preview or download server-generated API documentation PDF for a given API id.
User guide /guide This help document.
3. API List — browse and manage
  1. Open API List from the sidebar (default home).
  2. Use Search (name, code, URL, method), Status (active/inactive), and HTTP method to narrow the table. Click Refresh to reload from the server.
  3. View (eye icon): opens a read-only dialog with the full definition (request tree and response structure).
  4. Edit (pencil): opens API Builder with that API loaded for update.
  5. Test (play icon): opens API Tester with that API pre-selected.
  6. Delete (trash): asks for confirmation, then removes the definition from the backend.
4. API Builder — create or change a definition
  1. Go to API Builder. For a new API, open /builder without query parameters. To edit, use Edit from the list or open /builder?editId=1 (replace 1 with your API id).
  2. Basics: set API name, URL (base) (must be http:// or https://), optional Path template (must start with /), Method, Description, and Active.
  3. Path placeholders: use curly braces in the path, e.g. {fieldName}. The tester reads values from the JSON body using the same property names. Example: path /users/{userId} needs a top-level userId field in the request schema.
  4. Dynamic request fields: add rows for each JSON property. Choose a data type (STRING, INTEGER, BOOLEAN, OBJECT, ARRAY, etc.). Mark fields as mandatory if required. Optional sample / default values are merged when you test with an empty or partial body.
  5. Use Add field and nesting under OBJECT or ARRAY parents to model nested JSON. Sibling names must be unique at each level.
  6. Response body (optional): add output parameters and nested OBJECT/ARRAY fields so the PDF and view dialog document the expected JSON response.
  7. Click Save (create) or Update (edit). Fix any validation messages shown on the form.
 5. API Tester — validate and run
  1. Open API Tester or jump from a row’s Test action.
  2. Choose an API in Select API. The form builds inputs from the saved request field tree (or empty object if none).
  3. Validate: checks the JSON against the schema, applies defaults, and shows the resolved URL without calling the upstream system.
  4. Execute: performs the real HTTP call (GET/DELETE send query parameters from scalar body fields; POST/PUT/PATCH send the JSON body). Inactive APIs cannot be executed.
  5. Read the response status, body, and any error or timeout message shown below the actions.
 6. PDF Viewer — documentation export
  1. Open PDF Viewer.
  2. Enter the numeric API id (the same id shown in the list and tester). The viewer loads the PDF from {{ pdfBackendPathExample }} (replace the numeric id in the path).
  3. Use Download PDF to save the file, or rely on the embedded viewer controls.
  4. You can open the viewer with a pre-filled id using a URL like {{ pdfViewerQueryExample }} (change the id at the end).
7. Suggested workflow
  1. Define the API in API Builder → Save.
  2. Confirm it appears in API List and use View to verify fields.
  3. Run Validate then Execute in API Tester.
  4. Share PDF documentation from PDF Viewer for stakeholders.

Tip: sample JSON bodies for POST /api/create are in the project samples/ folder if you prefer importing definitions via the backend API or tools such as Postman.
